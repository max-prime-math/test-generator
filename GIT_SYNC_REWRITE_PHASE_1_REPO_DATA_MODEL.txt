Current Test Generator data model
---------------------------------
The app stores domain state directly:

- user questions in localStorage key "math-test-bank-v2"
- demo questions in localStorage key "math-test-demo-bank-v1"
- custom classes in localStorage key "math-test-custom-classes-v1"
- saved tests in localStorage key "tg-test-library-v1"
- draft config in localStorage key "tg-test-draft-v1"
- uploaded images in IndexedDB via src/lib/image-store.svelte.ts

This is not a repository working tree yet.

What to implement
-----------------
A. Introduce a repository worktree format

Create a module such as src/git/repoDataModel.ts that defines a stable file
layout for the app's syncable data. Use individual files where that improves
git diffs and reduces unnecessary conflicts.

Recommended layout:

- README.md
- manifest.json
- curriculum/custom-classes.json
- questions/index.json
- questions/<question-id>.json
- tests/index.json
- tests/<test-id>.json
- images/<stored-image-basename>

Guidelines:

- Do not include demo questions by default unless explicitly requested.
- Do not include transient render errors, preview artifacts, checked flags, or
  other noisy fields unless they are intentionally part of the user data.
- Do not include the current draft test config unless the user explicitly opts
  into tracking drafts later.
- Keep image bytes binary when possible. If a browser storage limitation forces
  base64, isolate that encoding in the repo data model module.
- Include a manifest with schema version, generatedAt, app version if available,
  and enough metadata to detect incompatible layouts.
- Treat imported repo entries as untrusted input. Validate schema, path,
  content type, file count, file size, total size, and image references before
  producing app data.
- Reject ambiguous paths, including absolute paths, backslashes, NUL bytes,
  encoded traversal, ".git" at any segment, case-insensitive duplicates,
  Unicode-normalization duplicates, reserved device names, and unexpected file
  extensions.
- Do not store credentials, OAuth client secrets, access tokens, API keys, repo
  remote config, or transient provider state in the repo data model.
- Do not introduce raw HTML rendering of imported repo content. If question text
  or rich content is rendered through a preview pipeline, it must not be able to
  execute JavaScript or fetch credential-bearing resources.

B. Add conversion helpers

Implement pure helpers:

- exportAppDataToRepoEntries(...)
- importRepoEntriesToAppData(...)
- normalizeRepoPath(...)
- isReservedRepoPath(...)

The exported entry type should support text and binary content:

type RepoDataEntry = {
  path: string;
  kind: "file";
  content: string | Uint8Array;
};

The import helper should validate all required files and produce a structured
result. Do not mutate localStorage directly inside these helpers.

C. Preserve app state ownership for now

In this phase, localStorage and IndexedDB remain the app's source of truth.
The repository worktree is a projection of that data. Do not introduce git
commits or remotes yet.

D. Add tests

Add focused tests for:

- exporting questions with choices, parts, narratives, graph metadata, and
  diagnostics
- exporting saved tests
- exporting custom classes
- path safety and reserved path rejection
- round-tripping exported data back into app data
- stable output ordering
- omission of noisy fields
- rejection of malicious remote entries, including traversal, .git paths,
  duplicate normalized paths, oversized blobs, unknown required schema versions,
  and image references with missing or unexpected targets
- imported rich/question content cannot create executable HTML/script paths

Non-goals
---------
- Do not implement git objects, commits, refs, index, staging, or remotes.
- Do not bring back the old sync panel.
- Do not bring back the old three-way conflict manager.
- Do not wire this into the UI yet except maybe behind tests or developer-only
  helpers.

Definition of done
------------------
1. Test Generator can produce a deterministic repository entry list from local
   app data.
2. The repo data model can reconstruct questions, custom classes, saved tests,
   and image references from entries.
3. Path validation blocks absolute paths, "..", NUL bytes, and ".git" paths.
4. Malicious or oversized repo entries are rejected before app state can be
   mutated.
5. No credentials or provider state are included in exported repo entries.
6. No old conflict-resolution UI or three-way merge code is reintroduced.
7. npm run check passes.
8. npm run regression passes if import/serialization behavior changed.
