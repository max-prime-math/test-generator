What to implement
-----------------
A. Multi-remote orchestration

Support a single bank repo syncing to multiple configured remotes, initially:

- GitHub
- Google Drive

Add explicit operations such as:

- Push to selected remote
- Fetch selected remote
- Pull selected remote
- Push to all configured remotes
- Fetch all configured remotes

Only add "sync all" if the behavior is precise and safe. A reasonable first
definition is:

1. project current app data into local repo
2. require clean/committed local state, or offer to commit first
3. fetch remotes
4. stop if any remote diverged from local in a way that cannot fast-forward
5. push the same local commit to each selected remote

Do not hide partial failures. If GitHub succeeds and Drive fails, report exactly
that and leave enough state for the user to retry Drive.
Track per-remote target commit, starting remote head/revision, final result, and
retry state. Do not mark the bank globally synced unless every selected remote
is at the intended commit. Do not let a retry against one remote re-push or
overwrite another remote that already succeeded.

B. Remove or quarantine obsolete sync code

Audit src/lib/sync and scripts/test-sync*. Remove old provider-sync code if it
is no longer used. If any modules remain only for Google Drive connection or
Drive remote transport, move or rename them so they do not imply old
backup/restore/conflict sync.

Remove dead references to:

- old provider fan-out backup
- GitHub Contents API sync
- legacy localStorage GitHub token storage
- legacy Google Drive backup/restore provider flow, except code deliberately
  adapted into the new Drive remote
- old ConflictModal/GistSyncPanel concepts
- timestamp/three-way app conflict manager
- auto-sync interval settings

E. Security/data-safety audit

Verify:

- tokens are not stored in repo data, commits, terminal output, errors, logs,
  Drive packages, or screenshots
- GitHub token persistence is explicit, and recommended tokens are
  fine-grained, expiring, repo-scoped, and least-privilege
- Google passwords are never requested, handled, stored, proxied, or logged
- Google Drive access tokens are memory-only and revoked/cleared where possible
- token-bearing network requests are allowlisted to GitHub/Google API origins
  only; no CORS proxy or user-provided API host is used
- docs/UI do not imply browser credential storage is secure against XSS or
  malicious same-origin scripts
- docs/UI warn that public GitHub repositories and broadly shared Drive folders
  can expose bank content
- .git cannot be represented as normal app data or user-editable synced data
- symlink/submodule/unsupported Git tree modes cannot become app data
- malicious remote paths, duplicate normalized paths, and unexpected file types
  are rejected before refs or app state change
- imported question/rich content cannot execute JavaScript through raw HTML or
  fetch credential-bearing resources
- deployment docs include the expected Content Security Policy or equivalent
  header posture for GitHub/Google API access and Google OAuth/Picker scripts
- remote/package size limits prevent memory and quota exhaustion
- failed remote operations leave local app data and repo refs recoverable
- no untrusted CORS proxy is used
- project data export does not include transient render/preview errors unless
  deliberately tracked
- image data is handled deterministically and safely
- multi-remote partial failures are visible and retryable

-------------------

- configure both GitHub and Google Drive for one bank repo
- confirm the app never asks for Google/GitHub account passwords
- confirm the app warns before syncing bank content to a public GitHub repo or
  broadly shared Drive folder
- make one local commit and push it to both remotes
- use a fresh browser/profile to restore from GitHub
- use a fresh browser/profile to restore from Google Drive
- force or simulate a case where GitHub push succeeds and Drive push fails,

- retry only the failed remote and confirm it succeeds without redoing the

- force or simulate a stale Drive package/head and confirm the app refuses to
  overwrite it
- confirm no remote silently overwrites another remote or local app data
- confirm no old sync settings, backup/restore controls, auto-sync options, or
  conflict modals are reachable

The question this QA should answer is: is multi-remote sync explicit,
retryable, and non-destructive?
