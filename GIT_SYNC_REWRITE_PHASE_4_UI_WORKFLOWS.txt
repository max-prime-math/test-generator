You are working in the test-generator repository at the repo root.

Goal of this phase
------------------
Expose the repo-backed local git and GitHub remote behavior through a simple,
honest Test Generator UI. Keep the workflow smaller than Typr's editor UI, but
use the same underlying architecture. Leave room in the UI model for additional
remotes, especially Google Drive, without adding Drive repo sync in this phase.

Important constraint
--------------------
Do not bring back the old sync provider panel or three-way conflict modals.
This UI should be git-oriented:

- local status
- commit current app data
- configure GitHub remote
- push
- fetch
- pull fast-forward only

What to implement
-----------------
A. Replace the placeholder sync button behavior

The header sync button should open a new Git/GitHub panel or modal. The current
Google Drive connection modal can either remain reachable from that panel or
stay as a separate secondary action.

The panel should show:

- current local branch
- working tree status summary
- last local commit
- configured remotes, if any
- upstream/ahead/behind summary, if implemented
- clear errors for dirty tree, missing token, bad token, or diverged history

B. Local commit workflow

Add controls to:

- project current app data into the repo working tree
- show changed file count
- commit all current app data with a user-supplied message
- show recent commits

Keep this honest. If app data has not been projected into the repo yet, say so.

C. GitHub remote setup

Add controls to:

- connect GitHub token
- choose or enter owner/repo
- select branch/default branch
- save non-secret remote config
- disconnect/clear token

Token handling must use the credentials module from Phase 3.
The UI must never ask for Google passwords or GitHub passwords. GitHub tokens
should use a password-style input and should not remain visible in the DOM after
connect. If persistent token storage exists, it must be an explicit opt-in, not
an accidental side effect of connecting.

The UI should not assume GitHub will forever be the only remote. It is fine for
the only working remote setup panel in this phase to be GitHub, but labels and
state shapes should not prevent a Google Drive remote from being added in Phase
5.
Make repository visibility and token scope visible enough that a user does not
accidentally push a private bank to a public GitHub repository.
Warn that pushed bank content may include proprietary or student-identifying
material if the selected GitHub repository is public or broadly shared.

D. Remote operations

Add controls for:

- fetch
- pull, fast-forward only
- push

Each operation must:

- show progress/busy state
- prevent duplicate concurrent operations
- use the per-repo operation lock from Phase 2
- show structured errors
- redact known secrets from all rendered errors and progress messages
- avoid blocking question editing and preview
- leave local state unchanged on failure

E. No merge/conflict UI yet

If pull or push detects divergence:

- stop
- explain that histories diverged
- show local and remote commit ids if available
- do not auto-resolve
- do not write conflict markers into app data
- do not resurrect the old three-way conflict picker

F. Tests

Add component/service tests where practical for:

- sync button opens git panel
- commit action calls local repo service
- push/pull buttons call remote service
- operation errors render clearly
- token inputs do not echo connected tokens back into the UI
- persistent token storage requires explicit opt-in, if supported
- no password fields or copy asks for Google/GitHub account passwords
- rendered errors redact GitHub and Google token-shaped strings
- diverged history message does not trigger app data mutation

Non-goals
---------
- No full terminal implementation.
- No old provider list.
- No backup/restore buttons.
- No Google Drive repo/file sync in this phase.
- No automatic periodic sync.

Human QA checkpoint
-------------------
After automated checks pass, ask the user to manually verify the UI workflow:

- open the sync button and confirm it shows the new Git/GitHub panel
- commit current app data through the UI
- configure GitHub credentials and remote settings through the UI
- confirm the UI asks for a GitHub token, not a GitHub password
- confirm the UI never asks for a Google password
- confirm token persistence, if available, is explicitly opted into
- confirm the UI warns before pushing bank content to a public or broadly shared
  remote
- fetch, pull, and push through the UI
- confirm buttons show busy/disabled states during operations
- confirm common errors are understandable, including missing token, bad token,
  dirty working tree, and diverged history
- confirm Google Drive, if visible, is labeled connection-only or clearly not
  part of repo sync yet
- confirm no old sync settings, auto-sync options, backup/restore controls, or
  conflict modals are reachable

The question this QA should answer is: does the UI explain the new model without
exposing dangerous actions?
