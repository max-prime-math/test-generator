Goal of this phase
------------------
Adapt Typr's browser local git backend to Test Generator so the repository data
model from Phase 1 can be committed as real git history in browser storage.

Important constraint
--------------------
This phase is local git only. Do not implement GitHub remotes yet. Do not bring
back app-level three-way conflict management.

What to implement
-----------------
A. Add a Test Generator git storage layer

Create modules such as:

- src/git/repoBackend.ts
- src/git/repoStore.ts
- src/git/pathFilters.ts
- src/git/repoDataBridge.ts

Use Typr's browser git backend as the reference architecture:

- real loose objects, trees, commits, refs, HEAD, config, and index
- browser-managed storage for .git internals
- .git hidden/reserved from normal app data paths
- structured RepoResult / RepoError style results

Keep direct git-object implementation details behind repoBackend.ts.
Add a per-repo operation lock so concurrent tabs, duplicate clicks, or
overlapping async operations cannot write refs, index, object storage, or app
projection state at the same time. Prefer the Web Locks API with a documented
fallback and timeout.

B. Define the Test Generator repository identity

For the first implementation, one Test Generator installation can own one local
repo. Store its git internals under a stable local repo id, for example:

- repo id: "test-generator-bank"
- git storage root: IndexedDB object store keyed by repo id
- working tree: generated from Phase 1 repo data entries

Do not store tokens in repo config or app data.
Do not store remote credentials, Google access tokens, GitHub PATs, OAuth client
secrets, or Drive package credentials anywhere in repo internals.

C. Bridge app data to the git working tree

Implement a bridge that:

- exports current app data to repo entries
- compares repo entries to current tracked/working tree state
- writes entry content to the git backend working tree representation
- detects local app changes since the last repo projection

Be explicit about write direction. In this phase, the app is still the source of
truth and git is the history layer. Importing repo data back into app state can
exist as a pure helper, but do not expose restore/pull workflows yet.
Any import helper must validate the complete repo snapshot and prepare a
recoverable app-data result before mutating localStorage or IndexedDB.

D. Keep remote-agnostic seams

Even though this phase has no remotes, do not design the local repo service as
GitHub-only. Phase 3 will add GitHub first, and Phase 5 will add Google Drive as
a second remote. Local refs, object storage, commit graph operations, and working
tree operations should be reusable by both.

E. Implement local commands/services

Expose a service API:

- initRepository
- status
- stageAll or stagePaths
- commit
- log
- getCurrentBranch
- listBranches
- createBranch
- switchBranch, if it can be safe with generated app data

At minimum, support "save current app data as a local commit":

1. project current app data into the working tree
2. stage all changed repo-data files
3. create a commit with a message

Use a deterministic default author if the user has not configured one:

- name: "Test Generator Local"
- email: "test-generator-local@example.invalid"

F. Add tests

Use memory-backed git storage where possible, modeled after Typr's
createMemoryGitFileStorage().

Test:

- repo init is idempotent
- concurrent repo operations are serialized or rejected safely
- first commit writes expected files
- status detects question/test edits
- committing after an edit creates a second commit
- binary image entries survive commit/readback
- .git and unsafe paths are rejected
- symlink/submodule/tree entries with unsupported modes are rejected if they can
  appear through the backend
- ref updates happen only after required objects are present
- repo state persists through the chosen browser storage abstraction where
  practical

Non-goals
---------
- No GitHub push/pull/fetch.
- No merge UI.
- No old sync provider panel.
- No three-way app conflict manager.
- No timestamp-based auto-resolution.
