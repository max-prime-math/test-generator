Important design decision
-------------------------
Google Drive is not a git server. Do not pretend it supports refs/objects over a
native git protocol. Choose and document one explicit storage strategy:

Recommended first strategy:

- store a compact repository bundle/snapshot file in the chosen Drive folder
- include enough data to reconstruct refs, commits, trees, blobs, and metadata
- treat the Drive file as the remote's authoritative repo package
- update it only after a successful local-to-remote transaction
- use compare-and-swap semantics based on the expected remote head, revision,
  checksum, or equivalent metadata; never overwrite a changed Drive package
  silently
- prefer temp upload plus verification before replacing or advancing the
  authoritative package/pointer

Alternative acceptable strategies:

- store loose object/ref files under a Drive folder if the provider can update
  them safely enough
- store pack-like chunks plus refs if large banks make a single bundle too
  expensive

Whichever strategy is chosen, keep it behind a remote provider interface so the
UI can treat GitHub and Google Drive as remotes with common operations where
possible.

Important conflict constraint
-----------------------------
Do not bring back the old three-way app conflict manager.

For this phase:

- push to Drive may require the Drive remote head to match the last fetched
  Drive head
- fetch from Drive imports the Drive repo package into local remote refs
- pull from Drive should be fast-forward-only
- divergence should stop with a clear non-destructive error

What to implement
-----------------
A. Define the Drive remote package format

Create a versioned format, for example:

- tg-drive-git-remote-v1.json or tg-drive-git-remote-v1.tgz

It should include:

- schema version
- remote id/name
- default branch
- refs
- objects or a compact object bundle
- created/updated timestamps
- app/repo metadata needed for validation

Avoid including credentials. Avoid including transient UI state.
Include integrity metadata that lets the importer verify object hashes and the
remote head before updating local remote refs. Set explicit limits for package
size, file count, path length, blob size, image size, commit graph depth, and
total imported bytes.

B. Add Drive remote provider service

Build on the existing Google Drive OAuth/folder picker code, but do not reuse
old backup/restore/conflict semantics.
Use Google Identity Services OAuth only. Do not ask for, handle, store, proxy,
or log Google passwords. Use the narrowest practical Drive scope, preferably
drive.file. Drive access tokens must remain memory-only; do not store refresh
tokens. Revoke/clear access tokens on disconnect where possible.

The Google Picker API key is public client configuration, not a password. If it
is configurable, document that it should be restricted in Google Cloud by HTTP
referrer and API.

Token-bearing requests must be allowlisted to Google Identity, Picker, Drive API,
and Drive upload origins. Do not support user-provided Drive API hosts, CORS
proxies, or token-bearing redirect URLs.

Implement common remote operations where possible:

- connect/configure Drive remote
- fetch
- push
- pull fast-forward-only
- inspect remote summary

C. Integrate with remote config

Allow the repo to have both:

- origin or github: GitHub remote
- drive or backup: Google Drive remote

Non-secret remote config may store Drive folder id/name/url and remote package
filename. OAuth tokens must remain in the credential/provider layer and must not
be committed or exported.
Drive folder ids and package filenames must be validated before storage and
displayed as text, not interpreted as HTML.

D. Update UI

Add Google Drive as a remote option in the Git/sync panel created in Phase 4.
It should be clear whether the user is operating on GitHub, Google Drive, or
both.

For this phase, one-at-a-time operations are acceptable:

- Push to GitHub
- Push to Google Drive
- Fetch from GitHub
- Fetch from Google Drive
- Pull from GitHub
- Pull from Google Drive

Do not implement "sync all remotes" yet unless the service layer is already
safe enough.

E. Add tests

Mock Google Drive API/file operations. Test:

- Drive package creation from local repo
- Drive fetch imports refs/objects without mutating local branch prematurely
- Drive fetch rejects invalid schemas, bad object hashes, unsafe paths,
  unsupported modes, oversized packages/blobs, and unexpected file types
- Drive push only updates the remote package after all data is ready
- Drive push refuses to overwrite when the remote revision/head/checksum changed
  since the last fetch
- temp upload or package replacement failure leaves the previous Drive package
  recoverable
- failed Drive upload leaves local refs/worktree unchanged
- fast-forward-only Drive pull succeeds
- diverged Drive pull stops safely
- credentials are not included in Drive package content
- Google access tokens are memory-only and are redacted from all errors
- token-bearing requests are sent only to allowlisted Google origins

Non-goals
---------
- No old provider fan-out backup.
- No timestamp merge/conflict resolution.
- No full merge UI.
- No automatic multi-remote sync yet, unless trivial and safe.

Human QA checkpoint
-------------------
After automated checks pass, ask the user to manually verify:

- connect Google Drive
- confirm the app uses Google's OAuth screen and never asks for a Google
  password
- select or create a Drive folder
- configure Google Drive as a repo remote
- push the local repo package/snapshot to Drive
- reload the app or use a fresh browser/profile, then connect the same folder
  and fetch/pull from Drive
- confirm GitHub still works independently after Drive is configured
- simulate Drive divergence or a failed upload if practical, then confirm the
  app stops safely and reports the issue clearly
- simulate a changed Drive package/head if practical, then confirm push refuses
  to overwrite it
- inspect exported/package data enough to confirm credentials are not included

The question this QA should answer is: can Drive act as a separate repo remote
without pretending to be GitHub?

Definition of done
------------------
1. Google Drive can be configured as a repo remote.
2. Local commits can be pushed to a Drive remote package.
3. Fetch imports Drive remote refs/objects.
4. Pull from Drive is fast-forward-only and safe.
5. GitHub remote behavior from Phase 3 still works.
6. No credentials are stored in repo/package data.
7. Google passwords are never requested or stored; Drive access tokens are
   memory-only.
8. Drive push has stale-head/revision protection and does not silently
   overwrite changed remote packages.
9. Malicious or oversized Drive packages are rejected before refs or app state
   change.
10. npm run check passes.
11. Drive remote tests pass.
